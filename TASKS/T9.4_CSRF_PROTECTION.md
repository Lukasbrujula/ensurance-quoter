# T9.4: CSRF Protection

**Priority:** Medium — prevents cross-site request forgery on mutation endpoints
**Estimate:** 45 min
**Phase:** 9 (Security Hardening)

---

## 9 Pillars

### 1. Objective
Add CSRF protection to all state-changing API endpoints. Prevent malicious websites from making authenticated requests on behalf of logged-in agents.

### 2. Scope
- Origin/Referer header validation on all POST/PUT/DELETE API routes
- Custom header requirement for AJAX requests
- Exclude webhooks (called by external services, not browsers)

### 3. Dependencies
- All API routes. Find them:
  ```bash
  find app/api -name "route.ts" | sort
  ```
- Current auth guard implementation:
  ```bash
  cat lib/middleware/auth-guard.ts
  ```
- Middleware file:
  ```bash
  cat middleware.ts
  ```

### 4. Implementation

**Approach: Origin validation + custom header**

CSRF attacks work because browsers automatically include cookies with cross-origin requests. Two defenses:

1. **Origin validation:** Check that the `Origin` or `Referer` header matches our domain. Browsers always send these headers and they cannot be spoofed by JavaScript.

2. **Custom header requirement:** Require `X-Requested-With: XMLHttpRequest` (or a custom header like `X-CSRF-Protection: 1`) on all mutation requests. Browsers block cross-origin requests with custom headers unless CORS allows it (and we don't allow arbitrary origins).

**Create CSRF middleware:**

```typescript
// lib/middleware/csrf.ts

const ALLOWED_ORIGINS = [
  process.env.NEXT_PUBLIC_APP_URL,
  process.env.APP_URL,
  'http://localhost:3000',       // Dev
  'http://localhost:3001',       // Dev alternate
].filter(Boolean);

export function validateCSRF(request: Request): { valid: boolean; reason?: string } {
  const method = request.method.toUpperCase();
  
  // Only validate state-changing methods
  if (['GET', 'HEAD', 'OPTIONS'].includes(method)) {
    return { valid: true };
  }

  // Check Origin header first (most reliable)
  const origin = request.headers.get('origin');
  if (origin) {
    const isAllowed = ALLOWED_ORIGINS.some(allowed => 
      origin === allowed || origin === new URL(allowed!).origin
    );
    if (!isAllowed) {
      return { valid: false, reason: `Origin ${origin} not allowed` };
    }
    return { valid: true };
  }

  // Fall back to Referer header
  const referer = request.headers.get('referer');
  if (referer) {
    try {
      const refererOrigin = new URL(referer).origin;
      const isAllowed = ALLOWED_ORIGINS.some(allowed =>
        refererOrigin === allowed || refererOrigin === new URL(allowed!).origin
      );
      if (!isAllowed) {
        return { valid: false, reason: `Referer origin ${refererOrigin} not allowed` };
      }
      return { valid: true };
    } catch {
      return { valid: false, reason: 'Invalid referer URL' };
    }
  }

  // No Origin or Referer — could be a direct API call (Postman, curl, etc.)
  // Check for custom header as fallback
  const csrfHeader = request.headers.get('x-csrf-protection');
  if (csrfHeader === '1') {
    return { valid: true };
  }

  // If in development, allow requests without headers
  if (process.env.NODE_ENV === 'development') {
    return { valid: true };
  }

  return { valid: false, reason: 'Missing Origin, Referer, and X-CSRF-Protection headers' };
}
```

**Apply to API routes:**

Two options for integration:

**Option A: Add to existing auth guard (preferred if auth guard is already called on every route):**
```typescript
// In auth-guard.ts, add CSRF check before auth check
import { validateCSRF } from './csrf';

export async function authGuard(request: Request) {
  // CSRF check first
  const csrf = validateCSRF(request);
  if (!csrf.valid) {
    console.warn('[CSRF] Blocked:', csrf.reason);
    return Response.json({ error: 'Forbidden' }, { status: 403 });
  }
  
  // Then existing auth logic...
}
```

**Option B: Add to middleware.ts (catches everything):**
```typescript
// In middleware.ts, add CSRF validation for /api/* routes
if (request.nextUrl.pathname.startsWith('/api/')) {
  const csrf = validateCSRF(request);
  if (!csrf.valid) {
    return NextResponse.json({ error: 'Forbidden' }, { status: 403 });
  }
}
```

**Exclude webhook endpoints:**
Webhooks are called by Telnyx, not browsers. They don't send Origin headers.

```typescript
const CSRF_EXEMPT_PATHS = [
  '/api/ai-agent/webhook',
  '/api/telnyx/',
  '/api/auth/callback',
];

// In CSRF check:
const isExempt = CSRF_EXEMPT_PATHS.some(path => 
  request.nextUrl.pathname.startsWith(path)
);
if (isExempt) return { valid: true };
```

**Update frontend fetch calls:**

If any client-side code uses `fetch()` directly (not through a library), add the custom header:

```typescript
// For any raw fetch calls to our API:
fetch('/api/whatever', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'X-CSRF-Protection': '1',  // Add this
  },
  body: JSON.stringify(data),
});
```

Check if the codebase uses a shared fetch wrapper — if so, add the header there once.

- [ ] `npx tsc --noEmit` passes

### 6. Failure Handling
- If CSRF blocks legitimate requests (e.g., from a mobile app or Postman in production), the custom header provides a bypass mechanism for non-browser clients
- If Origin header is missing on some browsers (rare in 2026), Referer header is the fallback
- If both are missing, the custom header check applies
- In development, CSRF is disabled to avoid blocking local testing

### 7. Guardrails
- Do NOT add CSRF tokens to forms (Next.js server actions already have built-in protection)
- Do NOT require CSRF on GET/HEAD/OPTIONS requests
- Do NOT block webhook endpoints
- Do NOT break existing API integrations (Telnyx, Deepgram callbacks)
- Keep the implementation simple — Origin validation is sufficient for most cases

### 8. Human Checkpoint
- Open browser dev tools, make a normal API call from the app. Should work.
- Use curl to POST to an API endpoint without Origin header. Should get 403 in production mode.
- Verify webhook endpoints still work without Origin headers.
