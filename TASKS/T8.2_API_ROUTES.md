**Priority:** Critical — UI tasks depend on these endpoints
**Estimate:** 1.5 hrs

## Context

Phase 7 built API routes at `/api/ai-agent/` for a single agent (create, toggle, delete). Phase 8 refactors these into a proper multi-agent CRUD API backed by the new `ai_agents` table, plus transcript storage and retrieval endpoints.

## 9 Pillars

### 1. Objective
Build REST API for managing multiple AI agents, storing/retrieving transcripts, and pulling usage statistics.

### 2. Scope
- CRUD for ai_agents (list, get, create, update, delete)
- Transcript storage (called by webhook after AI calls)
- Transcript retrieval (for the detail page viewer)
- Usage stats endpoint (aggregated call counts and minutes)
- Refactor existing Phase 7 webhook to target specific ai_agent

### 3. Dependencies
- T8.1 completed (ai_agents, ai_transcripts tables)
- Existing Phase 7 AI agent code — find actual file locations:

- Existing API patterns: auth guard, rate limiting, Zod validation (check `/api/settings/route.ts` or `/api/quote/route.ts` as references)

**Webhook refactor:**

Update the existing `/api/ai-agent/webhook` (or `/api/telnyx/ai-webhook`) to:
1. Identify which `ai_agent` handled the call (from telnyx_assistant_id in the payload)
2. Store transcript messages in `ai_transcripts` after fetching from Telnyx API
3. Update `ai_agents.total_calls` and `total_minutes` counters
4. Update `ai_agents.last_call_at`

### 5. Success Criteria
- [ ] `GET /api/agents` returns list of user's AI agents
- [ ] `GET /api/agents/[id]` returns agent details with recent calls and stats
- [ ] `POST /api/agents` creates agent in DB + Telnyx, returns the agent
- [ ] `PUT /api/agents/[id]` updates agent in DB + syncs to Telnyx
- [ ] `DELETE /api/agents/[id]` removes agent from DB + Telnyx
- [ ] `GET /api/agents/[id]/transcripts/[callId]` returns ordered transcript messages
- [ ] `GET /api/agents/usage` returns per-agent and total usage stats
- [ ] All endpoints auth-guarded with requireUser()
- [ ] All endpoints validate ownership (user can only access their own agents)
- [ ] Request bodies Zod-validated
- [ ] Rate limiting applied
- [ ] Webhook stores transcripts in ai_transcripts table
- [ ] `npx tsc --noEmit` passes

### 6. Failure Handling
- If Telnyx createAssistant fails during POST /api/agents, create DB row with `status: 'error'` and `telnyx_assistant_id: null`. Return the agent with an error message. User can retry from the UI.
- If Telnyx deleteAssistant fails during DELETE, still delete the DB row. Log the orphaned assistant ID.
- If transcript fetch fails after a call, the call record still exists in ai_agent_calls. Transcript can be fetched later.
- If usage aggregation is slow, add materialized view or caching later. For now, direct query is fine.

### 7. Guardrails
- Do NOT remove or break existing Phase 7 endpoints until the new ones are verified working
- Do NOT expose Telnyx API keys or assistant IDs in responses to the client
- Do NOT allow users to access other users' agents (RLS + application-level checks)
- Do NOT make transcript storage blocking in the webhook flow (fire-and-forget or async)
- Follow existing API route patterns exactly (auth guard, Zod, rate limit, error sanitization)

### 8. Human Checkpoint
- Test: Create agent via API, verify Telnyx assistant created
- Test: Update agent name, verify Telnyx assistant updated
- Test: Delete agent, verify cleanup
- Test: Call the AI agent, verify transcript stored in ai_transcripts
