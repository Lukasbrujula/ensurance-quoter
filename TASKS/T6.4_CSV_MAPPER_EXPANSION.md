# T6.4: CSV Upload Mapper — Recognize New Columns

**Priority:** Medium — agents import leads in bulk via CSV  
**Estimate:** 1 hr  
**Phase:** 6 (Lead Data Expansion + CRM)  
**Depends on:** T6.2 (types and schemas updated)

---

## Context

The CSV upload wizard (4-step: upload → map → preview → done) lets agents import lead lists in bulk. Currently it maps columns to the original limited field set. With the expanded schema, agents should be able to import richer CSVs that include address, DOB, occupation, income, etc. The mapper also needs to handle the variety of column header names agents might use in their purchased lead lists.

## Tasks

### 1. Add new fields to the column mapping options
- In the column mapping step of the CSV wizard, add the new fields as mapping targets:
  - Date of Birth
  - Address
  - City
  - Zip Code
  - Marital Status
  - Occupation
  - Income Range
  - Dependents
  - Existing Coverage
  - Notes

### 2. Auto-detection header aliases
- Lead lists from different vendors use different column headers. Add fuzzy matching/aliases:

| Target Field | Recognized Headers |
|---|---|
| date_of_birth | DOB, Date of Birth, Birthday, Birth Date, Birthdate |
| address | Address, Street, Street Address, Address Line 1, Addr |
| city | City, Town, Municipality |
| zip_code | Zip, Zip Code, ZIP, Zipcode, Postal Code, Postal |
| marital_status | Marital Status, Marital, Marriage Status, Status (marital) |
| occupation | Occupation, Job, Job Title, Title, Position, Employment |
| income_range | Income, Income Range, Salary, Salary Range, Annual Income, Household Income |
| dependents | Dependents, Deps, Number of Dependents, Children, Kids |
| existing_coverage | Existing Coverage, Current Coverage, Existing Insurance, Current Policy, Coverage in Force |
| notes | Notes, Comments, Remarks, Memo, Agent Notes |

- Also add aliases for existing fields if not already present (e.g., "First Name" + "Last Name" → name)

### 3. Value normalization on import
- **Marital Status**: Normalize variations → canonical values
  - "M", "Married", "married", "MARRIED" → `married`
  - "S", "Single", "single" → `single`
  - "D", "Divorced", "divorced" → `divorced`
  - "W", "Widowed", "widowed" → `widowed`
- **Income Range**: Map dollar amounts or ranges to canonical enum values
  - "$50,000" or "50000" → `50k_75k` (nearest bracket)
  - "75K-100K" or "$75,000 - $100,000" → `75k_100k`
  - If exact value, bucket into the correct range
- **Date of Birth**: Parse common date formats
  - MM/DD/YYYY, YYYY-MM-DD, MM-DD-YYYY, M/D/YY
  - Store as ISO date string
- **Zip Code**: Strip to 5 digits if 9-digit format with dash, validate format
- **Dependents**: Parse as integer, clamp 0-20

### 4. Preview step updates
- The preview step should show the new fields in the preview table
- If the preview table gets too wide, prioritize: Name, Email, Phone, State, Status, then scroll for rest
- Show normalization results (e.g., "Married" → `married`) so agents can verify

### 5. Import execution
- When the CSV is processed, include all mapped new fields in the Supabase insert
- New leads imported via CSV should have `status = 'new'` by default
- Unmapped fields are left null (not empty string)

---

## Success Criteria

- [ ] All 10 new fields appear as mapping options in the column mapper step
- [ ] Auto-detection correctly identifies common header aliases (test with at least 3 variations per field)
- [ ] Marital status, income range, DOB, and zip code values are normalized correctly
- [ ] Preview table shows new fields with normalized values
- [ ] Imported leads have correct values in all mapped new columns
- [ ] Unmapped new columns are null (not empty strings)
- [ ] All imported leads get `status = 'new'`
- [ ] Existing CSV import functionality unchanged for the original fields
- [ ] `npx tsc --noEmit` passes

## Dependencies

- T6.2 completed (types with new fields exist)
- CSV upload wizard components (find with: `grep -r "csv\|upload\|import" --include="*.tsx" components/leads/`)
- Column mapping logic (likely in the wizard component or a utility file)
- Supabase insert function for leads

## Guardrails

- Do NOT change the 4-step wizard flow (upload → map → preview → done)
- Do NOT modify how existing fields are mapped or imported
- Do NOT make any new column a required mapping — all optional
- Do NOT change the CSV parsing library
- Do NOT auto-map ambiguous headers — if "Status" appears, let the user choose (could be marital status or lead status)

## Failure Handling

- If a date can't be parsed, set to null and show a warning in preview ("Could not parse date: '13/25/1990'")
- If income value can't be bucketed, set to null with a preview warning
- If the preview table becomes too wide with all new columns, add horizontal scroll (reuse ScrollableTable from T2c if available)
- If the auto-detect maps incorrectly, the user can always override in the manual mapping step — auto-detect is a convenience, not a requirement
