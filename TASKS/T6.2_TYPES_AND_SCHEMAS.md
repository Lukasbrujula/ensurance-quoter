# T6.2: TypeScript Types + Zod Schemas Update

**Priority:** Critical — all UI tasks depend on these types  
**Estimate:** 30 min  
**Phase:** 6 (Lead Data Expansion + CRM)  
**Depends on:** T6.1 (migration complete)

## Context

After T6.1 adds columns to the database, the TypeScript types and Zod validation schemas need to match. Every component that touches lead data reads from these types, so they must be updated before any UI work begins.

## Tasks

### 1. Update Lead TypeScript interface
- Find the existing Lead type/interface (likely in `lib/types/` or within a store file)
- Add all new fields with appropriate TypeScript types:


### 2. Create enum/union types
```typescript
type LeadStatus = 'new' | 'contacted' | 'quoted' | 'applied' | 'issued' | 'closed';

type MaritalStatus = 'single' | 'married' | 'divorced' | 'widowed' | 'domestic_partner';

type IncomeRange = 'under_25k' | '25k_50k' | '50k_75k' | '75k_100k' | '100k_150k' | '150k_250k' | 'over_250k';
```

### 3. Update Zod validation schemas
- Find existing lead Zod schemas (for API validation, form validation, CSV import)
- Add new fields as optional with appropriate validators:

```typescript
const leadSchema = z.object({
  // ... existing fields ...
  
  date_of_birth: z.string().date().nullable().optional(),
  address: z.string().max(500).nullable().optional(),
  city: z.string().max(100).nullable().optional(),
  zip_code: z.string().regex(/^\d{5}(-\d{4})?$/).nullable().optional(),
  marital_status: z.enum(['single', 'married', 'divorced', 'widowed', 'domestic_partner']).nullable().optional(),
  occupation: z.string().max(200).nullable().optional(),
  income_range: z.enum(['under_25k', '25k_50k', '50k_75k', '75k_100k', '100k_150k', '150k_250k', 'over_250k']).nullable().optional(),
  dependents: z.number().int().min(0).max(20).nullable().optional(),
  existing_coverage: z.string().max(500).nullable().optional(),
  status: z.enum(['new', 'contacted', 'quoted', 'applied', 'issued', 'closed']).default('new'),
  status_updated_at: z.string().datetime().nullable().optional(),
  follow_up_date: z.string().datetime().nullable().optional(),
  follow_up_note: z.string().max(1000).nullable().optional(),
  notes: z.string().max(5000).nullable().optional(),
});
```

### 4. Update lead-store types
- If the Zustand lead-store has its own type definitions or partial types for form state, update those to include the new fields
- Ensure `intakeData` or equivalent form state type includes the new fields
- Default values for new fields: `null` for optional fields, `'new'` for status

### 5. Export new types
- Add `LeadStatus`, `MaritalStatus`, `IncomeRange` to barrel exports
- Ensure all components can import the new types

## Success Criteria

- [ ] `Lead` interface includes all 14 new fields with correct types
- [ ] `LeadStatus`, `MaritalStatus`, `IncomeRange` types are defined and exported
- [ ] Zod schema validates all new fields correctly
- [ ] Zip code validation accepts 5-digit and 9-digit formats
- [ ] All existing code that uses Lead type still compiles (new fields are optional)
- [ ] `npx tsc --noEmit` passes with zero errors
- [ ] Lead store default state includes new fields

## Dependencies

- Lead type definition (find with: `grep -r "interface Lead" --include="*.ts" --include="*.tsx"`)
- Lead Zod schema (find with: `grep -r "leadSchema\|z.object" --include="*.ts" lib/`)
- `stores/lead-store.ts` — Zustand store type definitions
- Barrel export files (`lib/types/index.ts` or similar)

## Guardrails

- Do NOT change existing field types — only add new ones
- Do NOT make existing optional fields required
- Do NOT remove any existing fields
- Do NOT modify component code — only type/schema files
- All new fields MUST be optional (except `status` with default) to maintain backward compatibility

## Failure Handling

- If the Lead type is defined in multiple places (interface + Zod infer), update both and ensure they match
- If `npx tsc --noEmit` shows errors in components that destructure Lead, it means a required field was added — make it optional
- If Zod schema is used for both API input and database output, you may need separate schemas (input = partial, output = full)
