# T6.8: Activity Timeline on Lead Detail

**Priority:** Medium — gives agents full history of interactions with a lead  
**Estimate:** 1.5 hrs  
**Phase:** 6 (Lead Data Expansion + CRM)  
**Depends on:** T6.6 (lead status exists for status change events)

---

## Context

When an agent opens a lead, they need to quickly see: what's happened with this person? When did I call them? What quote did I run? When did I change their status? Currently this context is scattered — call logs are in one place, quotes are in the database, status changes aren't tracked at all. An activity timeline consolidates all lead interactions into a single chronological feed.

## Tasks

### 1. Create activity_logs table
- New Supabase table: `activity_logs`
- Schema:

```sql
CREATE TABLE activity_logs (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  lead_id UUID NOT NULL REFERENCES leads(id) ON DELETE CASCADE,
  agent_id UUID NOT NULL REFERENCES auth.users(id),
  activity_type TEXT NOT NULL,  -- 'status_change', 'call', 'quote', 'enrichment', 'note', 'follow_up', 'lead_created', 'lead_updated'
  title TEXT NOT NULL,           -- Short description: "Status changed to Quoted"
  details JSONB,                 -- Flexible payload: { from: 'new', to: 'quoted' } or { carrier_count: 8, top_carrier: 'Foresters' }
  created_at TIMESTAMPTZ DEFAULT NOW()
);

-- RLS: only see your own leads' activities
ALTER TABLE activity_logs ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users can view activities for their leads"
  ON activity_logs FOR SELECT
  USING (agent_id = auth.uid());
CREATE POLICY "Users can insert activities for their leads"
  ON activity_logs FOR INSERT
  WITH CHECK (agent_id = auth.uid());

-- Index for fast lead-based queries
CREATE INDEX idx_activity_logs_lead_id ON activity_logs(lead_id, created_at DESC);
```

### 2. Define activity types and their display

| activity_type | Icon | Title Pattern | Details Payload |
|---|---|---|---|
| `lead_created` | UserPlus | "Lead created" | `{ source: 'manual' | 'csv_import' | 'ai_agent' }` |
| `status_change` | ArrowRight | "Status changed to [new status]" | `{ from: 'new', to: 'contacted' }` |
| `call` | Phone | "Outbound call — [duration]" | `{ direction: 'outbound', duration_seconds: 180, has_transcript: true }` |
| `quote` | Calculator | "Quote generated — [N] carriers eligible" | `{ carrier_count: 8, top_carrier: 'Foresters', coverage: '$250K', term: '20Y' }` |
| `enrichment` | Search | "Lead enriched via People Data Labs" | `{ fields_updated: ['city', 'occupation', 'income_range'] }` |
| `follow_up` | Clock | "Follow-up scheduled for [date]" | `{ date: '2026-03-01T14:00:00Z', note: 'Call about term quote' }` |
| `note` | MessageSquare | "Note added" | `{ text: 'Wife concerned about premium...' }` |
| `lead_updated` | Pencil | "Lead details updated" | `{ fields_changed: ['address', 'phone'] }` |

### 3. Timeline UI component
- Create `components/leads/activity-timeline.tsx`
- Visual: Vertical timeline with dots, connecting line, icons, timestamps
- Each entry shows:
  - Activity type icon (colored by type)
  - Title text
  - Relative timestamp ("2 hours ago", "Yesterday at 3:15 PM", "Feb 20, 2026")
  - Expandable details (click to see payload — e.g., call transcript link, quote details, enrichment fields)
- Most recent at top
- Pagination or "Load more" for leads with many activities (show last 20 by default)

### 4. Position in lead detail
- Place the timeline in a dedicated section of the lead detail view
- Options:
  - **Tab** within the lead detail (alongside existing tabs if any)
  - **Right panel section** below the call log viewer
  - **Collapsible section** at the bottom of the lead detail
- Choose based on existing lead detail layout — the timeline should be prominent but not push the form fields down too far

### 5. Emit activity events from existing actions
- **Status change** (T6.6): When status updates, insert an activity_log entry
- **Quote generation**: After `/api/quote` returns results for a lead, insert activity
- **Call completion**: After a Telnyx call ends, insert activity (supplement existing call_logs)
- **Enrichment**: After PDL enrichment is applied (T6.5), insert activity
- **Follow-up**: When a follow-up is scheduled (T6.7), insert activity
- **Lead creation**: On new lead insert (manual or CSV), insert activity
- **Lead update**: On lead detail save (if fields changed), insert activity

### 6. Activity log API
- Create or extend an API endpoint to:
  - `GET /api/activity-log/[leadId]` — Fetch activities for a lead (paginated, newest first)
  - `POST /api/activity-log` — Insert a new activity (used by other endpoints internally)
- Auth-guarded, UUID validated, rate-limited (consistent with existing API patterns)

- Lucide React icons (already installed)
- shadcn/ui: Card, Collapsible, Button, ScrollArea (all installed)

## Guardrails

- Do NOT modify existing call_logs table — activity_logs is a separate, parallel log
- Do NOT make activity logging blocking — if it fails, the primary action (status change, quote, etc.) should still succeed
- Do NOT store sensitive data in activity details (no full transcripts — just a link/reference)
- Do NOT add activity logging to every minor UI interaction — only meaningful business events
- Activity log inserts should be fire-and-forget where possible (don't await in the critical path)

## Failure Handling

- If activity log insert fails, log the error but don't surface it to the user — the primary action already succeeded
- If the timeline component is slow with many entries, ensure the API query uses the index and limit/offset pagination
- If placing the timeline in the lead detail is too cramped, start with a separate tab and iterate on placement later
- If emitting events from all sources in one phase is too much, prioritize: status_change + call + quote first, then add enrichment + follow_up + lead_updated in a follow-up
