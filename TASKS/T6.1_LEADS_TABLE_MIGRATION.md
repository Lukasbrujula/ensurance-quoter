# T6.1: Leads Table Migration — Add New Columns

**Priority:** Critical — everything in Phase 6 depends on this  
**Estimate:** 30 min  
**Phase:** 6 (Lead Data Expansion + CRM)

---

## Context

The current `leads` table has minimal fields: id, agent_id, name, email, phone, state, age, gender, tobacco, medical, DUI, coverage, term, source. Insurance agents working from purchased lead lists often have richer data (address, DOB, occupation, etc.), and they need somewhere to store it. This migration adds the columns that the rest of Phase 6 builds on.

## Tasks

### 1. Add personal/contact columns
- `date_of_birth` — DATE, nullable. More precise than the existing `age` integer. When DOB is present, age can be calculated; when only age is known, DOB stays null.
- `address` — TEXT, nullable. Street address line.
- `city` — TEXT, nullable.
- `zip_code` — TEXT, nullable (not integer — leading zeros in NE/NJ/etc zip codes).
- `marital_status` — TEXT, nullable. Values: 'single', 'married', 'divorced', 'widowed', 'domestic_partner'.

### 2. Add financial/professional columns
- `occupation` — TEXT, nullable. Free-text job title.
- `income_range` — TEXT, nullable. Values: 'under_25k', '25k_50k', '50k_75k', '75k_100k', '100k_150k', '150k_250k', 'over_250k'. Matches PDL inferred_salary ranges.
- `dependents` — INTEGER, nullable. Number of dependents (0+).
- `existing_coverage` — TEXT, nullable. Free-text description of existing life insurance (e.g., "$200K group through employer").

### 3. Add CRM workflow columns
- `status` — TEXT, NOT NULL, DEFAULT 'new'. Values: 'new', 'contacted', 'quoted', 'applied', 'issued', 'closed'. This powers the lead status workflow in T6.6.
- `status_updated_at` — TIMESTAMPTZ, nullable. When the status last changed.
- `follow_up_date` — TIMESTAMPTZ, nullable. Scheduled follow-up date/time.
- `follow_up_note` — TEXT, nullable. Note attached to the follow-up.
- `notes` — TEXT, nullable. General agent notes about the lead.

### 4. Create the migration
- Use Supabase SQL editor or migration file
- Migration must be additive only (ALTER TABLE ADD COLUMN) — no dropping or renaming existing columns
- All new columns are nullable (except `status` which has a default) so existing leads aren't affected
- Set `status = 'new'` for all existing leads that don't have a status

### 5. Verify RLS still works
- Existing RLS policy is `agent_id = auth.uid()` — adding columns doesn't affect this
- Confirm reads and writes still work with the new columns through the existing API patterns

## Migration SQL

```sql
-- Phase 6: Lead data expansion
ALTER TABLE leads ADD COLUMN IF NOT EXISTS date_of_birth DATE;
ALTER TABLE leads ADD COLUMN IF NOT EXISTS address TEXT;
ALTER TABLE leads ADD COLUMN IF NOT EXISTS city TEXT;
ALTER TABLE leads ADD COLUMN IF NOT EXISTS zip_code TEXT;
ALTER TABLE leads ADD COLUMN IF NOT EXISTS marital_status TEXT;
ALTER TABLE leads ADD COLUMN IF NOT EXISTS occupation TEXT;
ALTER TABLE leads ADD COLUMN IF NOT EXISTS income_range TEXT;
ALTER TABLE leads ADD COLUMN IF NOT EXISTS dependents INTEGER;
ALTER TABLE leads ADD COLUMN IF NOT EXISTS existing_coverage TEXT;
ALTER TABLE leads ADD COLUMN IF NOT EXISTS status TEXT NOT NULL DEFAULT 'new';
ALTER TABLE leads ADD COLUMN IF NOT EXISTS status_updated_at TIMESTAMPTZ;
ALTER TABLE leads ADD COLUMN IF NOT EXISTS follow_up_date TIMESTAMPTZ;
ALTER TABLE leads ADD COLUMN IF NOT EXISTS follow_up_note TEXT;
ALTER TABLE leads ADD COLUMN IF NOT EXISTS notes TEXT;

-- Backfill existing leads
UPDATE leads SET status = 'new' WHERE status IS NULL;
```

## Success Criteria

- [ ] All 14 new columns exist on the `leads` table
- [ ] Existing leads are unaffected (all new columns nullable except status with default)
- [ ] Existing leads have `status = 'new'` after backfill
- [ ] RLS still works — can only read/write own leads
- [ ] Can insert a new lead with the new columns via Supabase client
- [ ] Can insert a new lead WITHOUT the new columns (they default to null/new)

## Dependencies

- Supabase project access (SQL editor or migration CLI)
- `leads` table current schema (see CODEBASE_AUDIT.md section 7)

## Guardrails

- Do NOT drop or rename any existing columns
- Do NOT change existing column types
- Do NOT modify RLS policies
- Do NOT modify any application code in this task (that's T6.2+)
- All new columns MUST be nullable (except status with default) to avoid breaking existing inserts

## Failure Handling

- If a column already exists (from a prior attempt), `IF NOT EXISTS` handles it gracefully
- If the backfill UPDATE fails, it's non-critical — the DEFAULT 'new' handles future inserts, and existing rows can be backfilled manually
- If Supabase SQL editor isn't available, generate the migration as a `.sql` file in the project for manual execution
