### 1. Objective

Build the data layer and rewrite the coaching API to return structured, typed coaching cards instead of generic string hints. This is the foundation — if the AI returns garbage, the frontend doesn't matter.

Three deliverables:
1. **Medication lookup table** — ~100 commonly prescribed medications mapped to conditions and carrier eligibility

3. **Coaching API rewrite** — New system prompt with DISC framework + medication DB + life-event triggers, returning structured JSON cards

After this task, the coaching API should return clean, Zod-validated card objects when given a transcript. The frontend doesn't change yet — that's T11.1b.

#### A. Medication Lookup Table (`lib/data/medications.ts`)

```typescript
export interface MedicationEntry {
  name: string;           // lowercase generic name: "metformin"
  aliases: string[];      // brand names: ["glucophage", "fortamet", "riomet"]
  condition: string;      // "Type 2 Diabetes"
  carrierResults: {
    carrierId: string;    // must match IDs in carriers.ts (amam, moo, jh, foresters, etc.)
    result: 'accept' | 'decline' | 'conditional' | 'unknown';
    detail?: string;      // "Preferred Death Benefit", "if controlled", etc.
  }[];
  severity: 'low' | 'moderate' | 'high';
  // low = most carriers accept (levothyroxine, omeprazole)
  // moderate = splits carriers (metformin, SSRIs)
  // high = most carriers decline (insulin, HIV meds, chemo)
}
```

**Data sources for carrier results:**
- AMAM Term Made Simple (495 drugs): Chat 4 extraction. Format: Medication → Condition → ACCEPT/DECLINE/conditional
- MOO Term Life Express (119 drugs): Chat 4 extraction. Binary DECLINE list.
- John Hancock: Use medical conditions matrix — JH is the most lenient (accepts diabetes at Preferred DB, accepts bipolar, MS, schizophrenia)
- Foresters: Use medical conditions matrix — accepts asthma with steroids, Crohn's, epilepsy, fibromyalgia
- Other carriers: Mark as 'unknown' unless the medical conditions matrix has relevant condition data

**Export a fuzzy search helper:**
```typescript
export function findMedication(query: string): MedicationEntry | null {
  const q = query.toLowerCase().trim();
  return MEDICATIONS.find(med =>
    med.name === q ||
    med.aliases.some(a => a.toLowerCase() === q) ||
    med.name.includes(q) ||
    med.aliases.some(a => a.toLowerCase().includes(q))
  ) ?? null;
}
```

```typescript
export function buildMedicationContext(): string {
  // Generates a compact format:
  // metformin|glucophage|fortamet → Type 2 Diabetes → amam:decline, moo:decline, jh:accept(Preferred DB), foresters:accept
  // One medication per line, ~80-100 chars each
  // Target: under 4000 tokens for ~100 medications

```

This keeps the system prompt manageable. If the full context is too large, this function can be configured to return only high-severity medications or only the top 70.

#### E. Coaching API Rewrite (`app/api/coaching/route.ts`)

**What stays the same:**
- Auth guard
- Rate limiting
- 5-second timeout
- Error handling
- The caller (frontend) still sends transcript chunks at regular intervals
- POST method

**What changes:**
- System prompt (completely rewritten — see below)
- Response format: JSON mode with structured cards
- Response validation: Zod schema
- Return shape: `{ cards: CoachingCard[] }` instead of `{ hints: string[] }`

**System prompt structure:**

```
ROLE:
You are a real-time sales intelligence system for a life insurance agent on a 
live phone call. You analyze the conversation transcript and return structured 
coaching cards. The agent sees your cards on a side panel — they can only glance, 
so be concise.

CARD TYPES:
[Define all 4 card types with exact JSON schemas]

COMMUNICATION STYLE DETECTION (DISC):
[Two-axis model: pace × orientation]
[Four quadrants with verbal signatures]
[Detection rules: when to emit, confidence levels]
[Persuasion mapping per quadrant]

Return a style card ONCE after hearing 30+ seconds of CLIENT speech (not agent).
You may update it ONCE more after 3-5 minutes if evidence changes the classification.
If the client has barely spoken, do NOT force a style card.

MEDICATION DATABASE:
[Output of buildMedicationContext() — compressed lookup]

When ANY medication name appears in the transcript, return a medication card 
immediately. If you recognize a medication not in this database, still return a 
card with the condition name and mark all carriers as "unknown".

LIFE EVENT TRIGGERS:
[Keyword list from life-event-triggers.ts]

When the client mentions any of these life events, return a life_event card with 
the corresponding cross-sell suggestion and script.

COACHING TIPS:
Return a coaching_tip card only when you have a genuinely useful, non-obvious 
suggestion. Examples: a follow-up question the agent hasn't asked, an objection 
reframe, a closing technique matched to the detected style.

RULES:
- Return 0-3 cards per request. Quality over quantity. Empty array is fine.
- Never repeat the same card content across calls to this endpoint.
- Keep all text concise — the agent is mid-conversation.
- Scripts should sound natural, not robotic.
- Never include medical advice. You surface carrier eligibility data only.
- Never fabricate carrier results. If unsure, use "unknown".
- Respond ONLY with valid JSON matching the schema. No markdown, no explanation.
```

**Response parsing:**

```typescript
const parsed = CoachingResponseSchema.safeParse(JSON.parse(responseText));
if (!parsed.success) {
  // Fall back to empty cards
  return NextResponse.json({ cards: [] });
}
// Add IDs and timestamps to each card
const cards = parsed.data.cards.map(card => ({
  ...card,
  id: crypto.randomUUID(),
  timestamp: Date.now(),
}));
return NextResponse.json({ cards });
```

### 6. Failure Handling

- If OpenAI returns malformed JSON → return `{ cards: [] }`, log error
- If medication DB makes the prompt too large → reduce to 70 medications, log a warning
- If the model returns more than 3 cards → truncate to 3
- If a card has wrong type discriminator → filter it out, return remaining valid cards
- If the model hallucinates carrier results → the Zod schema catches it if the result isn't in the enum. For subtle hallucinations (wrong carrier for wrong medication), this is a tuning issue caught in T11.1c
