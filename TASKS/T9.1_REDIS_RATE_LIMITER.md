# T9.1: Redis Rate Limiter (Replace In-Memory Map)

**Priority:** High — current rate limiter is ineffective in serverless
**Estimate:** 1 hr
**Phase:** 9 (Security Hardening)

---

## 9 Pillars

### 1. Objective
Replace the in-memory Map-based rate limiter with Upstash Redis so rate limits actually work across serverless function instances on Vercel.

### 2. Scope
- Install `@upstash/ratelimit` and `@upstash/redis` packages
- Create a shared Redis-backed rate limiter utility
- Replace every usage of the current in-memory rate limiter
- Add environment variables for Upstash Redis connection
- Graceful fallback if Redis is unavailable (don't block requests, just log)

### 3. Dependencies
- Current rate limiter implementation. Find it:
  ```bash
  grep -rn "rateLimit\|rate_limit\|rateLimiter\|RateLimit\|Map()" --include="*.ts" lib/middleware/ lib/
  grep -rn "rateLimit\|rateLimiter" --include="*.ts" app/api/
  ```
- Every API route that uses the current rate limiter
- Upstash Redis account (free tier: 10K requests/day, sufficient for dev/MVP)

### 4. Implementation

**Install packages:**
```bash
bun add @upstash/redis @upstash/ratelimit
```

**Create the rate limiter utility:**
```typescript
// lib/middleware/rate-limit.ts (replace existing)
import { Ratelimit } from '@upstash/ratelimit';
import { Redis } from '@upstash/redis';

// Singleton Redis client
const redis = new Redis({
  url: process.env.UPSTASH_REDIS_REST_URL!,
  token: process.env.UPSTASH_REDIS_REST_TOKEN!,
});

// Pre-configured rate limiters for different use cases
export const rateLimiters = {
  // General API: 60 requests per minute
  api: new Ratelimit({
    redis,
    limiter: Ratelimit.slidingWindow(60, '1 m'),
    prefix: 'rl:api',
  }),

  // Auth endpoints: 10 attempts per minute (brute force protection)
  auth: new Ratelimit({
    redis,
    limiter: Ratelimit.slidingWindow(10, '1 m'),
    prefix: 'rl:auth',
  }),

  // AI/LLM endpoints: 20 requests per minute (cost control)
  ai: new Ratelimit({
    redis,
    limiter: Ratelimit.slidingWindow(20, '1 m'),
    prefix: 'rl:ai',
  }),

  // Webhooks: 100 per minute (higher limit, external callers)
  webhook: new Ratelimit({
    redis,
    limiter: Ratelimit.slidingWindow(100, '1 m'),
    prefix: 'rl:webhook',
  }),
};

// Helper that returns a standard rate limit check
export async function checkRateLimit(
  limiter: Ratelimit,
  identifier: string // IP address or user ID
): Promise<{ success: boolean; limit: number; remaining: number; reset: number }> {
  try {
    const result = await limiter.limit(identifier);
    return result;
  } catch (error) {
    // If Redis is down, allow the request (fail open)
    // Log the error for monitoring
    console.error('[Rate Limit] Redis error, failing open:', error);
    return { success: true, limit: 0, remaining: 0, reset: 0 };
  }
}
```

**Update every API route:**
Find all routes using the old rate limiter and swap to the new one:
```typescript
// Before (in-memory):
const limiter = new Map();
// ... custom logic

// After (Redis):
import { rateLimiters, checkRateLimit } from '@/lib/middleware/rate-limit';

const { success, remaining } = await checkRateLimit(rateLimiters.api, ip);
if (!success) {
  return Response.json({ error: 'Too many requests' }, { 
    status: 429,
    headers: { 'Retry-After': '60', 'X-RateLimit-Remaining': String(remaining) }
  });
}
```

**Extract IP address consistently:**
```typescript
function getClientIP(request: Request): string {

    || request.headers.get('x-real-ip')
    || '127.0.0.1';
}
```

### 5. Success Criteria
- [ ] `@upstash/redis` and `@upstash/ratelimit` installed
- [ ] New rate limiter utility created with tiered limits (api, auth, ai, webhook)
- [ ] Every API route using the old in-memory limiter is updated
- [ ] Auth endpoints use the stricter `auth` limiter (10/min)
- [ ] AI endpoints (chat, coaching, proactive) use the `ai` limiter
- [ ] Webhook endpoints use the `webhook` limiter
- [ ] Rate limit responses include `Retry-After` and `X-RateLimit-Remaining` headers
- [ ] Graceful fallback: if Redis is unavailable, requests are allowed (fail open with logging)
- [ ] Old in-memory rate limiter code is removed
- [ ] `npx tsc --noEmit` passes
- [ ] App still works without Redis env vars set (falls back to allowing all requests)

### 6. Failure Handling
- If Upstash is unreachable, fail open (allow request, log error). Never block users because Redis is down.
- If env vars are missing, log a warning at startup and disable rate limiting (same as dev mode).
- If the old rate limiter has a different interface than expected, adapt the replacement to match.

### 7. Guardrails
- Do NOT fail closed (block requests when Redis is down)
- Do NOT change any API route behavior other than the rate limiting mechanism
- Do NOT add Redis for anything other than rate limiting in this task
- Do NOT remove the `INTERNAL_API_SECRET` auth guard — rate limiting is separate from authentication
- This is the ONLY new dependency allowed in Phase 9
