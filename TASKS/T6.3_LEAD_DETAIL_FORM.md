# T6.3: Lead Detail Form Expansion — New Fields

**Priority:** High — agents need to view/edit expanded lead data  
**Estimate:** 1.5 hrs  
**Phase:** 6 (Lead Data Expansion + CRM)  
**Depends on:** T6.2 (types and schemas updated)

---

## Context

The lead detail view currently shows basic fields (name, email, phone, state, age, gender, etc.). With the expanded schema from T6.1/T6.2, the form needs new sections for personal details, financial/professional info, and a notes area. This is where agents manually enter or review lead data before quoting.

## Tasks

### 1. Add "Personal Details" section
- Position: Below existing contact info section
- Fields:
  - **Date of Birth** — Date picker (shadcn Calendar/DatePicker). When set, auto-calculate and display age. When only age is known (no DOB), keep the existing age field functional.
  - **Address** — Text input, full width
  - **City** — Text input, half width (left)
  - **Zip Code** — Text input with 5-digit validation, half width (right). Note: `state` already exists — city + zip + state gives full location.
  - **Marital Status** — Select dropdown: Single, Married, Divorced, Widowed, Domestic Partner

### 2. Add "Financial & Professional" section
- Position: Below Personal Details
- Fields:
  - **Occupation** — Text input (free-text job title)
  - **Income Range** — Select dropdown: Under $25K, $25-50K, $50-75K, $75-100K, $100-150K, $150-250K, $250K+
  - **Dependents** — Number input (0-20), with +/- stepper if the pattern exists elsewhere in the app
  - **Existing Coverage** — Textarea (e.g., "$200K group through employer, $100K individual term")

### 3. Add "Notes" section
- Position: Bottom of the form, before save button
- Fields:
  - **Notes** — Textarea, resizable, max 5000 chars, with character count display
- This is the agent's scratchpad for anything that doesn't fit structured fields

### 4. Form layout
- Follow existing form patterns in the codebase (React Hook Form + shadcn FormField)
- Sections use collapsible Card or Accordion pattern (consistent with existing sections like Medical History)
- Two-column grid on desktop (`grid-cols-2`), single column on mobile
- All new fields are optional — no required validation except format validation (zip code, DOB)
- Dirty state tracking should include new fields (unsaved changes guard)

### 5. Persist to database
- On save, include all new fields in the Supabase upsert
- On load, populate form with existing values from the lead record
- Handle null values gracefully (empty inputs, not "null" strings)

### 6. Manual lead creation dialog
- If there's a "Add Lead" dialog/modal in the leads list, add a subset of the new fields there too
- Keep the dialog focused — only add: DOB (or age), city, zip, marital status, occupation
- Full details can be entered in the lead detail view

## Success Criteria

- [ ] Lead detail form shows Personal Details, Financial & Professional, and Notes sections
- [ ] All new fields render with correct input types (date picker, select, text, number, textarea)
- [ ] Fields pre-fill when loading an existing lead with data in the new columns
- [ ] Fields save correctly to Supabase on form submit
- [ ] Empty/null fields save as null (not empty strings)
- [ ] Zip code validates as 5-digit or 5+4 format
- [ ] DOB auto-calculates displayed age when present
- [ ] Two-column layout on desktop, single column on mobile
- [ ] Dirty state tracking includes new fields
- [ ] "Add Lead" dialog includes subset of new fields
- [ ] `npx tsc --noEmit` passes

## Dependencies

- T6.2 completed (types and Zod schemas exist)
- Lead detail component (find with: `grep -r "lead-detail\|LeadDetail" --include="*.tsx" components/`)
- Lead form component (may be within lead detail or separate)
- Lead store — Supabase save/load functions
- "Add Lead" dialog component
- shadcn/ui: Input, Select, Textarea, Calendar/DatePicker, Card, FormField (all installed)

## Guardrails

- Do NOT remove or reorder existing form fields
- Do NOT change how existing fields save to the database
- Do NOT make new fields required in the form
- Do NOT modify the quote intake form (that's separate from the lead detail form)
- Do NOT break the existing dirty state / unsaved changes guard
- Follow existing form patterns exactly — don't introduce new form libraries or patterns

## Failure Handling

- If DatePicker component isn't installed or working, fall back to a standard date input (`<Input type="date" />`)
- If the lead detail form uses a different pattern than React Hook Form (e.g., controlled inputs with useState), adapt to the existing pattern rather than introducing RHF
- If the form becomes too long, group sections into an Accordion that defaults to collapsed for the new sections (personal details, financial), keeping the original fields always visible
