# T9.2: Service Role to Auth Client Migration

**Priority:** High — defense in depth for data access
**Estimate:** 1.5 hrs
**Phase:** 9 (Security Hardening)

---

## 9 Pillars

### 1. Objective
Migrate server actions and data layer functions from the Supabase service role client (which bypasses RLS) to the auth client (which respects RLS). This ensures that even if there's a bug in application-level auth checks, RLS prevents data leakage.

### 2. Scope
- Audit every file that uses the service role client
- Determine which can safely switch to auth client
- Migrate each one, preserving the agent_id filtering
- Keep service role ONLY where absolutely necessary (webhook handlers from external services, migrations)
- Document which files still use service role and why

### 3. Dependencies
- Supabase client files. Find them:
  ```bash
  grep -rn "createServiceRoleClient\|supabaseAdmin\|service_role\|SUPABASE_SERVICE_ROLE" --include="*.ts" lib/ app/
  grep -rn "createAuthBrowserClient\|createServerClient\|auth-server\|auth-client" --include="*.ts" lib/ app/
  ```
- Understand the two client types:
  - **Service role client:** Uses `SUPABASE_SERVICE_ROLE_KEY`. Bypasses ALL RLS. Full database access.
  - **Auth client (server):** Uses session cookies. Queries respect RLS (agent_id = auth.uid()).
- RLS policies on all tables (see CODEBASE_AUDIT.md section 7)

### 4. Implementation

**Step 1: Audit every service role usage**

Run the grep above and categorize each usage:

| File | Current Client | Can Migrate? | Reason |
|---|---|---|---|
| Server actions (leads, settings) | Service role | YES | User is authenticated, session available |
| API routes with session auth | Service role | YES | Cookie-based session available in request |
| Webhook endpoints (Telnyx AI) | Service role | NO | No user session — external caller |
| Background jobs / cron | Service role | NO | No user context |

**Step 2: Create auth-aware server client helper**

If one doesn't already exist, create a helper that builds a Supabase client from the request cookies:

```typescript
// lib/supabase/auth-server.ts (may already exist)
import { createServerClient } from '@supabase/ssr';
import { cookies } from 'next/headers';

export function createAuthServerClient() {
  const cookieStore = cookies();
  return createServerClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!,
    {
      cookies: {
        getAll() { return cookieStore.getAll(); },
        setAll(cookiesToSet) {
          cookiesToSet.forEach(({ name, value, options }) => {
            cookieStore.set(name, value, options);
          });
        },
      },
    }
  );
}
```

**Step 3: Migrate data layer functions**

For each function that currently uses service role:

```typescript
// BEFORE (bypasses RLS):
import { supabaseAdmin } from '@/lib/supabase/admin';

export async function getLeads() {
  const user = await requireUser();
  const { data } = await supabaseAdmin
    .from('leads')
    .select('*')
    .eq('agent_id', user.id); // Application-level filter only
  return data;
}

// AFTER (RLS enforced):
import { createAuthServerClient } from '@/lib/supabase/auth-server';

export async function getLeads() {
  const supabase = createAuthServerClient();
  const { data } = await supabase
    .from('leads')
    .select('*');
  // No .eq('agent_id', ...) needed — RLS handles it
  // But keeping it is fine as belt-and-suspenders
  return data;
}
```

**Step 4: Keep service role only where justified**

These places MUST keep service role:
- **Webhook handlers** (`/api/ai-agent/webhook`, `/api/telnyx/*`) — No user session. External services call these.
- **Internal API calls** with `INTERNAL_API_SECRET` — Server-to-server, no cookies.
- **Database migrations** — Admin operations.
- **User creation side effects** — If creating data for a user who just signed up (before their first authenticated request).

Add a comment to each service role usage explaining why:
```typescript
// SECURITY: Service role required — this endpoint is called by Telnyx webhook, no user session available
const supabase = createServiceRoleClient();
```

**Step 5: Verify RLS covers all tables**

Confirm every table has RLS enabled and policies match:
```sql
SELECT tablename, rowsecurity FROM pg_tables WHERE schemaname = 'public';
-- All should show rowsecurity = true

SELECT tablename, policyname, cmd, qual FROM pg_policies WHERE schemaname = 'public';
-- Verify agent_id = auth.uid() or equivalent on every table
```

Check the newer tables from Phases 6-8 (activity_logs, ai_agents, ai_transcripts, ai_agent_calls) have RLS.

### 5. Success Criteria
- [ ] Every server action that has user context uses auth client (not service role)
- [ ] Every API route with session auth uses auth client
- [ ] Service role usage is limited to: webhooks, internal API, migrations
- [ ] Each remaining service role usage has a comment explaining why
- [ ] RLS is confirmed enabled on ALL tables (including Phase 6-8 tables)
- [ ] All existing functionality still works (leads, quotes, settings, agents, calls)
- [ ] No data leaks: user A cannot see user B's data even if application filter is removed
- [ ] `npx tsc --noEmit` passes

### 6. Failure Handling
- If a function breaks after migration (RLS blocks a query that used to work), the RLS policy may be missing or wrong. Fix the policy, don't revert to service role.
- If a server action can't access cookies (running in a context without request), it may need to stay on service role with a documented reason.
- If the auth client factory doesn't exist yet, create it following Supabase SSR documentation for Next.js App Router.

### 7. Guardrails
- Do NOT remove the service role client entirely — it's still needed for webhooks and admin operations
- Do NOT remove application-level agent_id filters when switching to auth client — keep both (belt and suspenders)
- Do NOT change RLS policies in this task — only verify they exist
- Do NOT change any user-facing behavior
- Test each migrated function before moving to the next

### 8. Human Checkpoint
- Create two test accounts. Log in as User A, create a lead. Log in as User B, verify you cannot see User A's lead. This should already work, but verify after migration.
- Check Supabase dashboard: RLS enabled on every table.
