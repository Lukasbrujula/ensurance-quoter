# T10.6: Notifications Inbox — Bell Icon + Slide-Out Panel

**Priority:** High — agents need to know when AI collects leads and callbacks are due
**Estimate:** 1.5 hrs
**Phase:** 10 (Dashboard + UX Polish)

## 9 Pillars

### 1. Objective
Add a notifications system with a bell icon in the top nav (right side, next to profile button). Clicking opens a slide-out panel showing AI agent collected leads, overdue follow-ups, and system alerts. Unread count badge on the bell.

### 2. Scope
- Bell icon in top nav with unread count badge
- Slide-out sheet (right side, like a drawer) with notification list
- Notification types: AI agent lead, overdue follow-up, scheduled callback approaching, status change
- Mark as read (individual and "mark all read")
- Click notification to navigate to relevant lead/page
- Notifications table in database (or derive from existing data)

### 3. Dependencies
- Top nav component (find: `grep -rn "TopNav\|top-nav" --include="*.tsx" components/navigation/`)
- Activity logs from Phase 6 (potential data source)
- AI agent calls from Phase 7-8
- Follow-up data from Phase 6
- shadcn/ui: Sheet, Badge, Button, ScrollArea, Separator

### 4. Implementation

**Approach decision: Dedicated notifications table vs derived from existing data**

Option A (recommended): **Derive notifications from existing data**
- No new table needed
- Query recent activity_logs + overdue follow-ups + AI agent calls
- "Read" state stored in a simple `notification_reads` table or JSONB on agent_settings

Option B: **Dedicated notifications table**
- More flexible but more infrastructure
- Use if notification types will expand beyond what existing data covers

**Go with Option A for MVP.** Create a lightweight tracking mechanism for read state.

**Read state storage:**
Add a column to agent_settings:
```sql
ALTER TABLE agent_settings ADD COLUMN IF NOT EXISTS last_notifications_read_at TIMESTAMPTZ DEFAULT NOW();
```
Any activity/event after this timestamp is "unread." When user opens notifications, update this timestamp. Simple and effective.

// components/navigation/notification-bell.tsx
'use client';

export function NotificationBell() {
  const [open, setOpen] = useState(false);
  const { data: notifications } = useNotifications();
  const unreadCount = notifications?.filter(n => !n.read).length || 0;

  return (
    <>
      <Button variant="ghost" size="icon" className="relative" onClick={() => setOpen(true)}>
        <Bell className="h-5 w-5" />
        {unreadCount > 0 && (
          <span className="absolute -top-1 -right-1 h-5 w-5 rounded-full bg-destructive text-destructive-foreground text-xs flex items-center justify-center">
            {unreadCount > 9 ? '9+' : unreadCount}
          </span>
        )}
      </Button>
      
      <Sheet open={open} onOpenChange={setOpen}>
        <SheetContent side="right" className="w-[400px] sm:w-[540px]">
          <SheetHeader>
            <div className="flex items-center justify-between">
              <SheetTitle>Notifications</SheetTitle>
              {unreadCount > 0 && (
                <Button variant="ghost" size="sm" onClick={markAllRead}>
                  Mark all read
                </Button>
              )}
            </div>
          </SheetHeader>
          <NotificationList notifications={notifications} onClose={() => setOpen(false)} />
        </SheetContent>
      </Sheet>
    </>
  );
}
```

function NotificationItem({ notification, onClick }) {
  return (
    <button
      onClick={onClick}
      className={cn(
        "w-full flex items-start gap-3 p-4 hover:bg-muted/50 text-left transition-colors",
        !notification.read && "bg-muted/30"
      )}
    >
      <div className={cn("mt-0.5 p-2 rounded-full", notification.iconBg)}>
        <notification.icon className="h-4 w-4" />
      </div>
      <div className="flex-1 min-w-0">
        <p className={cn("text-sm", !notification.read && "font-medium")}>
          {notification.message}
        </p>
        <p className="text-xs text-muted-foreground mt-1">
          {formatRelativeTime(notification.createdAt)}
        </p>
      </div>
      {!notification.read && (
        <div className="h-2 w-2 rounded-full bg-primary mt-2" />
      )}
    </button>
  );
}
```

**API endpoint:**
`GET /api/notifications` — Returns aggregated notifications:
1. Query activity_logs for recent events (last 7 days)
2. Query leads for overdue follow-ups
3. Query leads for upcoming callbacks (next 1 hour)
4. Query ai_agent_calls for recent AI leads
5. Compare timestamps against `last_notifications_read_at` for read state
6. Sort by timestamp, return last 50

**Polling for new notifications:**
Simple approach: poll every 60 seconds when the app is open:
```typescript
const { data: notifications } = useSWR('/api/notifications', fetcher, {
  refreshInterval: 60000, // Poll every minute
});
```

- [ ] `npx tsc --noEmit` passes

### 6. Failure Handling
- If notifications API fails, show error in sheet with retry
- If polling fails, silently retry next interval (don't show error toast every minute)
- If a notification links to a deleted lead, show "Lead not found" on navigation

### 7. Guardrails
- Do NOT use WebSockets for real-time notifications (polling is fine for MVP)
- Do NOT create a complex notifications table (derive from existing data)
- Do NOT add sound or browser push notifications (too invasive for MVP)
- Do NOT show notifications from before the agent's account creation
- Keep the notification list lightweight — max 50 items, older ones age out

- Open notifications, mark all read, verify badge clears
- Click a notification, verify it navigates correctly
