# T6.6: Lead Status Field + Workflow

**Priority:** High — transforms the CRM from a static list to a pipeline  
**Estimate:** 1 hr  
**Phase:** 6 (Lead Data Expansion + CRM)  
**Depends on:** T6.2 (types with LeadStatus exist)

---

## Context

Currently leads are a flat list with no workflow state. Insurance agents need to track where each lead is in the sales pipeline: did I contact them? Did I quote them? Did they apply? A status field with visual pipeline indicators turns the basic CRM into a usable sales tool.

## Tasks

### 1. Define the pipeline stages

| Status | Label | Color | Description |
|---|---|---|---|
| `new` | New | Blue | Fresh lead, not yet contacted |
| `contacted` | Contacted | Yellow/Amber | Agent has reached out (call, email) |
| `quoted` | Quoted | Purple | Quote has been generated and shared |
| `applied` | Applied | Orange | Application submitted to carrier |
| `issued` | Issued | Green | Policy issued — sale complete |
| `closed` | Closed | Gray | Lead closed — not interested, ineligible, or lost |

### 2. Status badge component
- Create a reusable `LeadStatusBadge` component
- Uses shadcn Badge with color variants matching the table above
- Shows the status label text
- Compact enough for table rows, detail headers, and cards

### 3. Status in leads list table
- Add a "Status" column to the leads CRM table
- Shows the `LeadStatusBadge` for each lead
- Column should be sortable and filterable (filter by status)
- Position: After Name column, before Email/Phone

### 4. Status filter on leads list
- Add a status filter dropdown/pills above the leads table (or extend existing filter bar)
- Options: All, New, Contacted, Quoted, Applied, Issued, Closed
- Multi-select allowed (e.g., show "New" + "Contacted" only)
- Default: show all except "Closed" (agents usually want active pipeline)

### 5. Status change in lead detail
- In the lead detail view, show current status prominently (top of page, near lead name)
- Add a status dropdown or horizontal pipeline stepper to change status
- Pipeline stepper visual (optional but impactful): shows all stages as connected dots/steps, current stage highlighted, completed stages filled
- On status change:
  - Update `status` in Supabase
  - Set `status_updated_at` to current timestamp
  - Auto-transitions: When a quote is generated (via the quote engine), offer to auto-advance status from "new" → "quoted" or "contacted" → "quoted"

### 6. Auto-status advancement
- When agent generates a quote for a lead, if lead status is "new" or "contacted", show a toast: "Move lead to 'Quoted'?" with a one-click action
- When agent makes a call to a lead (Telnyx), if lead status is "new", auto-advance to "contacted" (no confirmation needed — calling IS contacting)
- These are suggestions, not forced — agent can always manually set status

## Success Criteria

- [ ] Leads list table shows color-coded status badge for every lead
- [ ] Status column is sortable
- [ ] Status filter allows selecting one or more statuses to view
- [ ] Default filter excludes "Closed" leads
- [ ] Lead detail view shows current status prominently
- [ ] Agent can change status from lead detail (dropdown or stepper)
- [ ] Status change updates `status` and `status_updated_at` in Supabase
- [ ] Auto-advance prompt appears after generating a quote (if status is new/contacted)
- [ ] Auto-advance to "contacted" happens after making a call to a "new" lead
- [ ] `npx tsc --noEmit` passes

## Dependencies

- T6.2 completed (LeadStatus type exists)
- Leads list component (find with: `grep -r "lead-list\|LeadList\|leads-table" --include="*.tsx" components/`)
- Lead detail component
- Lead store — Supabase update functions
- shadcn/ui: Badge, Select, DropdownMenu (all installed)

## Guardrails

- Do NOT modify the quote engine or API
- Do NOT force status changes — auto-advance is always optional or easily reversible
- Do NOT remove the ability to set any status from any other status (no strict linear progression enforced)
- Do NOT change the leads table sort defaults — just add status as a sortable option
- Do NOT add a new API endpoint for status changes — use the existing lead update mechanism

## Failure Handling

- If the pipeline stepper visual is too complex, fall back to a simple Select dropdown for status change
- If auto-advance toast is annoying, it can be dismissed and the behavior can be toggled in preferences later
- If status filter breaks the existing search/sort, ensure filters compose correctly (search + status filter + sort all work together)
