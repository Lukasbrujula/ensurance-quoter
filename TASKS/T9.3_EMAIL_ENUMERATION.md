# T9.3: Email Enumeration Protection

**Priority:** Medium — prevents attackers from discovering valid email addresses
**Estimate:** 30 min
**Phase:** 9 (Security Hardening)

---

## 9 Pillars

### 1. Objective
Ensure login, registration, and password reset endpoints return identical error messages regardless of whether the email exists. Prevents attackers from probing for valid agent email addresses.

### 2. Scope
- Login form/API: same error for wrong email vs wrong password
- Registration: same behavior for existing vs new email
- Password reset: same response for existing vs non-existing email
- Auth callback: no information leakage on confirmation

### 3. Dependencies
- Auth components. Find them:
  ```bash
  grep -rn "signIn\|signUp\|resetPassword\|error\|Invalid" --include="*.tsx" components/auth/
  grep -rn "signIn\|signUp\|resetPassword" --include="*.ts" app/auth/ lib/
  ```
- Supabase Auth error messages (check what Supabase returns for different failure modes)

### 4. Implementation

**Login form:**
```typescript
// BEFORE (leaks info):
if (error.message.includes('Invalid login credentials')) {
  setError('Invalid email or password');
}
if (error.message.includes('Email not confirmed')) {
  setError('Please check your email to confirm your account');
}

// AFTER (generic):
// ALL auth failures return the same message
const GENERIC_AUTH_ERROR = 'Invalid email or password. If you recently signed up, check your email for a confirmation link.';

if (error) {
  setError(GENERIC_AUTH_ERROR);
}
```

**Registration:**
```typescript
// BEFORE (leaks info):
// Supabase may return "User already registered" for existing emails

// AFTER:
// Always show the same success message regardless of whether the email existed
const REGISTRATION_SUCCESS = 'If this email is not already registered, you will receive a confirmation link shortly.';

// Don't distinguish between "new account created" and "email already exists"
// Show the success message in both cases
```

**Password reset:**
```typescript
// BEFORE (leaks info):
// Different messages for "email sent" vs "email not found"

// AFTER:
// Always show success message
const RESET_SUCCESS = 'If an account exists with this email, you will receive a password reset link shortly.';
// Return this regardless of whether Supabase found the email
```

**Supabase-level protection:**

Check if Supabase has a setting for this. In the Supabase dashboard:
- Authentication > Settings > Enable "Confirm email" (prevents knowing if signup worked)
- Some Supabase versions return generic errors by default; verify behavior.

**Timing attack protection:**

Even with generic messages, different response times can leak info (existing email takes longer to hash password). Add a small random delay:

```typescript
// Add 100-300ms random delay to auth responses
await new Promise(resolve => setTimeout(resolve, 100 + Math.random() * 200));
```

### 5. Success Criteria
- [ ] Login returns same error message for wrong email vs wrong password vs unconfirmed email
- [ ] Registration returns same success message whether email is new or already exists
- [ ] Password reset returns same success message whether email exists or not
- [ ] No Supabase-specific error messages leak to the client
- [ ] Auth responses include random delay to prevent timing attacks
- [ ] Existing users can still log in, register, and reset passwords normally
- [ ] `npx tsc --noEmit` passes

### 6. Failure Handling
- If Supabase returns error types we need to distinguish (e.g., rate limiting vs auth failure), handle rate limiting separately with its own message, but keep all auth errors generic.
- If the auth components use different error handling patterns, normalize them all to the same approach.

### 7. Guardrails
- Do NOT change the actual authentication logic (signIn, signUp, resetPassword calls)
- Do NOT remove error handling — just make messages generic
- Do NOT break the login/register/reset flows
- Do NOT log user emails in error messages (this would leak to server logs)
- Keep the user experience friendly — generic doesn't mean unhelpful

### 8. Human Checkpoint
- Try logging in with a non-existent email. Note the error message.
- Try logging in with a real email and wrong password. Note the error message.
- Both messages should be identical.
- Try password reset with a non-existent email. Should show same success message as a real email.

### 9. Estimated Time
30 minutes
