## Context

The PDL (People Data Labs) enrichment pipeline already exists: there's an API endpoint (`/api/enrichment`), a UI popover (`lead-enrichment-popover.tsx`), and enrichment results are stored in the `enrichments` table. What's missing is the last mile â€” when enrichment returns data, it should auto-fill the expanded lead fields rather than just displaying results in an accordion.

## Tasks

### 1. Map PDL fields â†’ lead fields
Define the mapping from PDL response fields to lead database columns:

| PDL Field | Lead Column | Transform |
|---|---|---|
| `birth_year` | `date_of_birth` | Convert year to approximate DOB: `{birth_year}-06-15` (mid-year estimate). If `birth_date` is available, use exact date. |
| `location_street_address` | `address` | Direct copy |
| `location_locality` | `city` | Direct copy |
| `location_region` | `state` | Only update if lead's state is empty â€” don't overwrite agent-entered state |
| `location_postal_code` | `zip_code` | Take first 5 digits |
| `job_title` | `occupation` | Direct copy |
| `inferred_salary` | `income_range` | Map PDL ranges to our enum: "very high" â†’ 'over_250k', "high" â†’ '150k_250k', "mid" â†’ '75k_100k', etc. |
| `sex` | `gender` | Only update if lead's gender is empty |

### 2. Auto-fill on enrichment success
- After a successful PDL enrichment call, present the agent with a confirmation before overwriting:
  - Show a summary of what will be filled: "PDL found: DOB 1988, Portland OR 97201, Software Engineer, $100-150K"
  - Checkboxes next to each field â€” all checked by default
  - "Apply to Lead" button that writes selected fields to the lead
  - "Dismiss" to keep enrichment data stored but not applied
- This prevents accidentally overwriting agent-entered data

### 3. Update the enrichment popover
- The existing `lead-enrichment-popover.tsx` shows enrichment results in an accordion
- Add an "Apply to Lead" section at the bottom of the accordion that shows the field mapping
- When the agent clicks "Apply", update the lead form fields in the UI (dirty state) AND save to Supabase
- Fields that already have values in the lead show a âš ď¸Ź "Will overwrite: [current value] â†’ [PDL value]" warning

### 4. Handle the gender/age special cases
- `age`: If PDL provides `birth_year` and the lead only has an `age` integer, calculate and set both `date_of_birth` (approximate) and update `age` to match
- `gender`: PDL uses "male"/"female" â€” map to whatever format the existing gender field uses
- `state`: PDL's `location_region` is a full state name â€” ensure it matches the format in the state dropdown (abbreviation vs full name)

### 5. Track enrichment source
- When fields are populated from PDL, optionally store a metadata note in the activity timeline (T6.8) or in the enrichments table: "Fields populated from PDL enrichment on [date]: DOB, city, occupation, income"
- This helps agents know which data came from enrichment vs manual entry

## Guardrails

- Do NOT modify the PDL API call itself â€” only how results are applied to leads
- Do NOT auto-apply enrichment without agent confirmation
- Do NOT overwrite existing non-null values without showing the overwrite warning
- Do NOT change the enrichments table schema
- Do NOT store PDL raw data in the leads table â€” it stays in the enrichments table, only mapped fields go to leads

## Failure Handling

- If PDL returns fields in unexpected format, gracefully skip that field (don't crash the apply flow)
- If a mapped value fails validation (e.g., zip code is non-US format), skip that field and show a note
- If the lead save fails after applying enrichment, show error toast and revert the form to pre-enrichment state
- If PDL returns no usable fields for mapping, show "No additional lead data found in enrichment results"
