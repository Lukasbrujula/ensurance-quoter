# T9.5: Password Policy Hardening

**Priority:** Medium — GLBA compliance expects reasonable password requirements
**Estimate:** 30 min
**Phase:** 9 (Security Hardening)

---

## 9 Pillars

### 1. Objective
Enforce stronger password requirements on the registration and password reset forms. Add client-side validation with real-time feedback and server-side enforcement.

### 2. Scope
- Password complexity rules on registration form
- Same rules on password reset form
- Real-time strength indicator as user types
- Server-side Zod validation as final enforcement
- Breach detection via k-Anonymity check against HaveIBeenPwned API (optional, stretch goal)

### 3. Dependencies
- Auth form components:
  ```bash
  grep -rn "password\|Password" --include="*.tsx" components/auth/
  ```
- Zod schemas for auth forms:
  ```bash
  grep -rn "password.*z\.\|z\.string.*min" --include="*.ts" --include="*.tsx" components/auth/ lib/
  ```
- Supabase Auth password settings (check dashboard: Authentication > Settings)

### 4. Implementation

**Password requirements (GLBA-appropriate for financial services):**
- Minimum 10 characters (up from typical 8)
- At least 1 uppercase letter
- At least 1 lowercase letter
- At least 1 number
- At least 1 special character (!@#$%^&*()_+-=[]{}|;:',.<>?)
- Not the user's email address
- Maximum 128 characters (prevent denial-of-service via extremely long passwords)

**Zod schema:**
```typescript
const passwordSchema = z.string()
  .min(10, 'Password must be at least 10 characters')
  .max(128, 'Password must be less than 128 characters')
  .regex(/[A-Z]/, 'Password must contain at least one uppercase letter')
  .regex(/[a-z]/, 'Password must contain at least one lowercase letter')
  .regex(/[0-9]/, 'Password must contain at least one number')
  .regex(/[^A-Za-z0-9]/, 'Password must contain at least one special character');
```

**Password strength indicator component:**

Create a small reusable component that shows requirements being met in real time:

```typescript
// components/auth/password-strength.tsx

interface PasswordRequirement {
  label: string;
  met: boolean;
}

function getRequirements(password: string): PasswordRequirement[] {
  return [
    { label: 'At least 10 characters', met: password.length >= 10 },
    { label: 'Uppercase letter', met: /[A-Z]/.test(password) },
    { label: 'Lowercase letter', met: /[a-z]/.test(password) },
    { label: 'Number', met: /[0-9]/.test(password) },
    { label: 'Special character', met: /[^A-Za-z0-9]/.test(password) },
  ];
}

// Display: checklist with green checks / gray circles as user types
// Only show when password field is focused or has content
```

Keep it minimal — a small list below the password input with check/circle icons. Not a strength bar or score (those are misleading).

**Apply to both forms:**
- `components/auth/register-form.tsx` (or equivalent) — New account creation
- `components/auth/set-password-form.tsx` (or equivalent) — Password reset

**Supabase-side:**
Supabase Auth has its own minimum password length setting. Update it in the dashboard to match:
- Authentication > Settings > Minimum password length: 10

Note: Supabase doesn't enforce complexity rules server-side (only length). Our Zod validation is the primary enforcement. If a user somehow bypasses the client form, the API should also validate.

**Breach check (stretch goal):**

Use HaveIBeenPwned's k-Anonymity API to check if a password has appeared in known breaches. This is safe — only the first 5 characters of the SHA-1 hash are sent:

```typescript
async function isPasswordBreached(password: string): Promise<boolean> {
  const hash = await sha1(password);
  const prefix = hash.substring(0, 5);
  const suffix = hash.substring(5).toUpperCase();
  
  const response = await fetch(`https://api.pwnedpasswords.com/range/${prefix}`);
  const text = await response.text();
  
  return text.split('\n').some(line => line.startsWith(suffix));
}
```

If breached, show a warning: "This password has appeared in a data breach. Choose a different password for better security." Don't block, just warn (blocking creates too much friction).

### 5. Success Criteria
- [ ] Registration form enforces all 5 complexity rules
- [ ] Password reset form enforces same rules
- [ ] Real-time strength indicator shows which requirements are met
- [ ] Zod schema validates password complexity on form submission
- [ ] Form shows specific unmet requirements as validation errors
- [ ] Existing users with weak passwords can still log in (don't lock them out)
- [ ] Supabase minimum password length updated to 10
- [ ] `npx tsc --noEmit` passes

### 6. Failure Handling
- If HaveIBeenPwned API is unreachable (stretch goal), skip the breach check silently
- If Supabase dashboard setting can't be changed, the client-side + API validation is still the primary gate
- If existing password reset flow has a different component structure, adapt the strength indicator to fit

### 7. Guardrails
- Do NOT lock out existing users with weak passwords (enforcement is only on new passwords)
- Do NOT store password requirements in the database (they're code-level policy)
- Do NOT show the password strength indicator as a gamified bar — use a simple checklist
- Do NOT block form submission on breach check (just warn)
- Keep the UX friendly — help users create strong passwords, don't frustrate them

### 8. Human Checkpoint
- Try registering with "password" — should be rejected with specific feedback
- Try registering with "Password1!" (9 chars) — should fail on length
- Try "Password1!x" (11 chars) — should pass all checks
- Verify the strength indicator updates in real time

### 9. Estimated Time
30 minutes (45 if implementing breach check)
