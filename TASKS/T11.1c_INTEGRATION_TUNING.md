### 4. Implementation

#### A. Wire Coaching API → Card Stack

Find where the coaching API is currently called during live calls (likely in `ai-assistant-panel.tsx` or a hook). The existing pattern is:

```typescript
// OLD: send transcript, get string hints
const response = await fetch('/api/coaching', { body: { transcript } });
const data = await response.json();
// data.hints is string[] — displayed directly
```

Change to:

```typescript
// NEW: send transcript, get typed cards
const response = await fetch('/api/coaching', { body: { transcript } });
const data = await response.json();
// data.cards is CoachingCard[]
for (const card of data.cards) {
  if (card.type === 'style') {
    callStore.updateStyleCard(card);
  } else {
    callStore.addCoachingCard(card);
  }
}
```

**Deduplication:** The coaching API is called every ~15-30 seconds. The model might return the same medication card or life-event card again if the transcript still contains the same content. Before adding a card, check if a card with the same type + key content already exists:

```typescript
function isDuplicate(newCard: CoachingCard, existingCards: CoachingCard[]): boolean {
  return existingCards.some(existing => {
    if (existing.type !== newCard.type) return false;
    switch (newCard.type) {
      case 'style': return true; // only one style card, always replace
      case 'medication': return existing.type === 'medication' && 
        existing.medicationName.toLowerCase() === newCard.medicationName.toLowerCase();
      case 'life_event': return existing.type === 'life_event' && 
        existing.event === newCard.event;
      case 'coaching_tip': return false; // tips can repeat with different content
    }
  });
}
```

#### B. Transcript Context Management

The coaching API receives transcript chunks. Important decision: **send the full accumulated transcript or just the latest chunk?**

Recommendation: Send the **last 2-3 minutes of transcript** (not the full call). This keeps the prompt manageable and focuses the model on recent speech. The full transcript is too long for a 5-second timeout after a 15-minute call.

Also include a `previousCards` summary so the model knows what it's already surfaced:

```typescript
const body = {
  transcript: getRecentTranscript(180), // last 3 minutes
  previousCards: callStore.coachingCards.map(c => ({
    type: c.type,
    // minimal identifier so model doesn't repeat
    ...(c.type === 'medication' ? { medication: c.medicationName } : {}),
    ...(c.type === 'life_event' ? { event: c.event } : {}),
    ...(c.type === 'style' ? { quadrant: c.quadrant } : {}),
  })),
  callDuration: callStore.duration, // helps model decide when to emit style card
};
```

#### C. Call Log Persistence

When the call ends and the call log is saved, serialize coaching cards into the existing `coaching_hints` JSONB column:

```typescript
// In the call-end handler, before POST /api/call-log:
const coachingData = {
  // Summary fields for easy querying
  style_detected: styleCard?.quadrant ?? null,
  style_confidence: styleCard?.confidence ?? null,
  medications_detected: medicationCards.map(c => c.medicationName),
  life_events_detected: lifeEventCards.map(c => c.event),
  tips_count: tipCards.length,
  // Full card data for historical reference

};

// Save as coaching_hints (existing JSONB column)
await fetch('/api/call-log', {
  body: { ...callLogData, coaching_hints: coachingData }
});
```

No schema change — the `coaching_hints` column already accepts JSONB.

#### D. Backward Compatibility

Any component that previously read `coachingHints` (the old string array from the store) needs to handle both formats gracefully. The cleanest approach:

- Keep the old `coachingHints: string[]` in the store
- The new `coachingCards: CoachingCard[]` is the primary source
- If any UI still reads `coachingHints`, check if it's still needed. If it's only the call log viewer showing old call data, it should handle both the old string[] format and the new structured format in the JSONB.

#### E. Prompt Tuning (if needed)

After wiring everything, test with real transcript data. Common issues to watch for:

1. **Style card too early** — Model emits style card from agent's speech, not client's. Fix: add to prompt "Only classify the CLIENT's style, not the agent's. The agent is the one asking questions."

2. **Medication false positives** — Model detects "I'm taking care of my mother" as a medication. Fix: add to prompt "Only flag actual medication names, not casual uses of words that happen to match."

3. **Too many cards** — Model returns 3 cards every 15 seconds, flooding the panel. Fix: add to prompt "Return cards sparingly. An empty array is perfectly fine. Only surface genuinely new information."

4. **Style card confidence too high** — Model gives 4/4 confidence after 30 seconds. Fix: add to prompt "Confidence 4 requires 5+ minutes of clear behavioral evidence. After 1 minute, max confidence is 2. After 3 minutes, max is 3."

5. **Cards too verbose** — Agent note or tips are paragraph-length. Fix: add to prompt "Maximum: 15 words per tip, 20 words per agent note."

These are small prompt edits, not code changes. Document what you changed and why.

### 5. Success Criteria

- [ ] During a simulated call, coaching cards appear in the panel from live transcript data
- [ ] Style card appears after ~60 seconds of client speech
- [ ] Medication card appears within one coaching cycle (~15-30s) of medication being mentioned
- [ ] Life-event card appears within one coaching cycle of trigger phrase
- [ ] Duplicate cards are not added (same medication mentioned twice doesn't create two cards)
- [ ] Cards persist in the call log after call ends
- [ ] Call log viewer displays the coaching summary from the new JSONB format
- [ ] Old call logs with string[] coaching_hints still display correctly
- [ ] The coaching API stays under 5 seconds per call
- [ ] No orphaned state — clearing cards on new call works
- [ ] `npx tsc --noEmit` passes
- [ ] `bun run build` passes

### 6. Failure Handling

- If coaching API returns error → don't add cards, don't crash the call
- If coaching API times out (5s) → skip this cycle, try again next interval
- If response has invalid cards mixed with valid → add only the valid ones
- If call log save fails → cards are lost (ephemeral) — acceptable for MVP
- If transcript is empty or too short → don't call coaching API (min 50 chars)

### 7. Guardrails

- Do NOT modify Telnyx WebRTC, Deepgram, or SIP code
- Do NOT modify Chat Mode
- Do NOT add new database columns — use existing coaching_hints JSONB
- Do NOT increase coaching API call frequency beyond existing rate (every 15-30s)
- Do NOT send full call transcript — cap at last 3 minutes
- Remove any hardcoded sample cards from T11.1b testing before merging

### 8. Human Checkpoint

**End-to-end test:**

1. Start a call (or simulate with dev tools)
2. Speak as the "client":
   - Start slow, mention family: "I want to make sure my grandkids are taken care of..."
   - Mention medication: "I take lisinopril for my blood pressure"
   - Mention life event: "My daughter just had her first baby"
3. Verify:
   - Style card appears (should be S — slow, family-focused)
   - Medication card appears for lisinopril
   - Life-event card appears for "New Grandchild"
   - Cards are visually correct (right colors, right content)
   - No duplicate cards
4. End the call
5. Check the call log — verify coaching_hints contains the card data
6. Open an old call log — verify it still displays correctly (backward compat)

**Tuning test:**

If cards are wrong (wrong style, wrong medication, too many, too few), adjust the system prompt and retest. Document changes. This is iterative — 2-3 rounds of prompt adjustment is normal.
